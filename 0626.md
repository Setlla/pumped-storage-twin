# 抽蓄土石方平衡数字化管理系统

## 一、项目理解与总体定位

本项目面向抽水蓄能工程建设期土石方平衡管理场景，目标不是单纯建设三维展示大屏，而是围绕开挖、运输、填筑、中转、弃渣、进度、料质和现场修正，形成一套可管理、可展示、可计算、可预警、可追溯的数字化管理系统。

抽水蓄能工程土石方量大、周期长、参与方多，土石方平衡关系直接影响施工组织、运输效率、料源利用、弃渣控制和工程进度。系统建设应以“时间进度 + 空间流向 + 现场修正 + 调配辅助”为主线，先形成1.0验证版，再根据现场数据、硬件条件和业务反馈逐步扩展生产闭环。

### 1.1 系统定位

系统定位为面向工程建设期的土石方平衡数字化管理平台，以时空关系管理为核心，建立三维可视化、轻量级数据接入与治理、业务规则分析、辅助调配、预警处置和报表输出的一体化应用。

| 定位维度 | 具体说明 |
|---|---|
| 业务定位 | 服务土石方开挖、运输、填筑、中转、弃渣、动态调配、进度优化和风险预警，重点管理时间进度与空间流向的匹配关系。 |
| 技术定位 | 三维可视化 + 轻量级数据接入与治理底座 + 业务规则与预警 + 数据报表。 |
| 管理定位 | 支撑项目内部管理、分包运输管控、料源利用率提升、施工进度辅助决策和现场问题闭环。 |
| 战略定位 | 形成可沉淀源码、数据标准、业务规则和后续复制能力的数字化系统样板。 |

### 1.2 建设判断

1. 1.0阶段不宜一次性铺开完整生产级闭环，应先建设可演示、可验证、可迭代的首期版本。
2. 系统不能只做前端页面或三维效果，必须配套项目级轻量数据接入与治理底座。
3. 土石方平衡的核心不是单点算量，而是施工时间、空间流向、料质约束、现场揭露和调配决策之间的动态闭环。
4. 对实时三维模型更新、全自动智能调配、复杂地质自动建模等能力，应采用分阶段建设和风险分级管理。
5. 1.0阶段应以内管实用、基础功能和数据闭环为优先，展示效果服务于业务表达。

## 二、建设目标与1.0范围边界

### 2.1 总体建设目标

建设一套覆盖土石方平衡核心过程的数字化管理系统，实现基础资料统一管理、施工动态数据采集、工程量统计分析、车辆运输监管、动态调配辅助、进度优化、风险预警和数据报表输出。

1. 数据可管：基础资料、工程量、车辆、道路、料质、模型、测量成果等数据能够统一管理、导入、校验、追溯。
2. 态势可看：通过驾驶舱、GIS/三维场景和图表看清土石方总体态势、调配流向、进度偏差和风险状态。
3. 方案可算：基于供需、料质、道路、时间窗口、容量等约束输出可解释、可复核的调配建议。
4. 风险可报：形成进度偏差、供需不平衡、容量超限、料质不匹配、路线约束冲突等异常风险清单和报表。
5. 成果可沉淀：形成源码、数据库、接口、部署和操作文档，为知识产权、后续迭代和多项目复制提供基础。

### 2.2 1.0建设抓手

1.0建议以“一舱、一库、一底座、一算法、一表”为建设抓手，先形成基础闭环。

| 抓手 | 含义 | 1.0可交付成果 |
|---|---|---|
| 一舱 | 可视化大屏 / 数字孪生驾驶舱 | 集成指标看板、GIS/三维时空场景、调配流向和风险态势。 |
| 一库 | 土石方基础数据台账 | 统一管理挖方、填方、堆场/渣场/中转场、道路、车辆、设备、料质、施工阶段等数据。 |
| 一底座 | 项目级轻量数据接入与治理底座 | 统一数据口径、编码、坐标、单位、版本、权限、接口和追溯机制。 |
| 一算法 | 调配方案计算模型 | 输出分时段调配建议、关键指标、路线/流向、方量、车次/成本估算和异常提示。 |
| 一表 | 调配报表和风险清单 | 支撑工程人员复核、内部沟通、阶段汇报和后续优化。 |

### 2.3 1.0范围边界

| 范围类型 | 1.0建议边界 | 说明 |
|---|---|---|
| 必须实现 | 驾驶舱、三维一张图、基础资料、关键要素、工程量统计、车辆定位、数据导入、预警中心、报表、权限管理、设计参考数据与施工实测数据管理。 | 支撑系统真实可用和阶段汇报。 |
| 建议实现 | 称重数据、抬斗识别、无人机成果管理、道路匹配矩阵、动态调配基础版、进度优化基础版、地质模型与料源动态修正基础能力。 | 根据数据条件和接口条件逐步落地。 |
| 后续迭代 | 全自动智能调配、实时三维模型自动重建、全量点云在线处理、全厂商硬件即插即用、完整视频AI平台。 | 作为2.0及后续迭代方向，避免1.0范围失控。 |
| 必须预留 | 国产化适配、多项目复制、数据接口扩展、设备接入扩展、局级平台对接。 | 架构层面预留，不要求1.0完全实现。 |

## 三、时空关系驱动的业务主线

土石方平衡调配不是静态的一次性配对，而是“分区、分层、分时段”的持续匹配过程。系统要同时管住时间关系和空间关系，才能把调配建议算得准、解释得清、执行得下去。

### 3.1 时空关系的核心含义

| 管理内容 | 主要事项 | 管理结果 |
|---|---|---|
| 时间管理 | 开挖开始/结束时间、填筑启用/完成时间、中转启用时间、弃渣接纳时间、施工阶段计划、实际进度。 | 明确各阶段什么时候该完成什么事。 |
| 空间管理 | 开挖区、坝体填筑区、中转场、弃渣场、道路通道的位置关系。 | 明确料从哪里来、到哪里去。 |
| 料质管理 | 料源属性、坝体填筑要求、禁配关系、优先级。 | 明确什么料能用、什么料不能用。 |
| 路径管理 | 道路可达、运距、坡度、通行限制、限制时段。 | 明确怎么运、走哪条路。 |
| 容量管理 | 中转场容量、弃渣场容量、填筑区需求量、阶段性存量。 | 明确能不能放、还能放多少。 |
| 预警管理 | 时间冲突、空间冲突、料质冲突、容量冲突、进度偏差。 | 明确哪里失衡、如何纠偏。 |

### 3.2 关键空间对象与调配关系

| 空间对象 | 关键属性 | 对应关系 | 系统管控重点 |
|---|---|---|---|
| 开挖区/料场 | 位置、方量、料源属性、分层、可开挖时间。 | 对应坝体填筑区 / 中转场 / 弃渣场。 | 先判定可利用性，再判定最近可达去向。 |
| 坝体填筑区 | 坝段、分区、高程、设计需求量、料质要求、计划填筑时间。 | 对应一个或多个开挖区/料场。 | 按施工阶段和料质要求接收合格料源。 |
| 中转场 | 位置、容量、允许料质、停留周期。 | 承接阶段性错峰料源，再转运至坝体填筑区。 | 解决时间不匹配，不作为长期堆存场。 |
| 弃渣场 | 位置、容量、许可边界、可弃料类型。 | 承接不可利用或阶段性超量料源。 | 严格控制容量和运距，避免可利用料误弃。 |
| 施工道路 | 通行能力、距离、限制条件、时段管控。 | 连接上述各空间对象。 | 作为运距、车次、时效和成本测算基础。 |

### 3.3 时空关系支撑模块

| 支撑模块 | 主要作用 | 关键输出 |
|---|---|---|
| 基础资料管理 | 管理对象、属性、编码。 | 时空对象台账。 |
| 进度计划管理 | 管理施工阶段、时间窗口、计划/实际进度。 | 分阶段供需计划。 |
| 空间关系与路网管理 | 管理对象落位、道路矩阵、去向关系。 | 路径与可达关系。 |
| 料源料质匹配 | 管理料源属性、坝体需求、禁配规则。 | 可利用料源清单。 |
| 容量约束管理 | 管理中转场、弃渣场、填筑区容量。 | 容量预警与去向限制。 |
| 调配计算 | 综合供需、料质、路径、时间、容量约束。 | 分阶段调配建议。 |
| 预警纠偏 | 识别时间、容量、道路、料质冲突。 | 异常清单与处置建议。 |

### 3.4 调配计算输入与输出

调配计算应以规则型、可解释、可人工确认为原则。

```text
挖方供给
  + 填方需求
  + 料质约束
  + 道路距离
  + 时间窗口
  + 堆场容量
  + 运输能力
    -> 调配计算
    -> 分时段调配建议 + 路线/流向 + 方量 + 车次/成本估算 + 异常提示
```

1.0优先实现基础调配计算，定位为辅助决策和方案复核工具，不作为完全替代人工调度的自动决策系统。计算结果应至少包含分阶段来源去向关系、是否直达坝体或进入中转/弃渣、推荐路径、方量分配、车次与成本估算。

## 四、总体技术架构设计

系统采用分层架构设计，划分为数据源与感知层、数据接入层、轻量数据接入与治理底座、业务服务层、应用功能层、用户展示层和基础设施层。分层设计的目的是避免前端直接绑定各类硬件和第三方系统，提升系统可维护性、可扩展性和后续复制能力。

| 架构层级 | 核心内容 | 建设重点 |
|---|---|---|
| 用户与展示层 | 驾驶舱、三维一张图、大屏展示、业务管理端、后续移动端。 | 强调可视化、可汇报、可操作。 |
| 应用功能层 | 基础资料、关键要素、动态调配、进度优化、预警中心、数据报表、系统管理。 | 围绕业务闭环组织功能。 |
| 业务服务层 | 工程量统计、运输轨迹分析、调配计算、预警规则、报表生成、权限审计。 | 将业务逻辑沉淀为服务能力。 |
| 轻量数据底座 | 数据接入、清洗、统一编码、坐标统一、数据关联、主题库、标准接口。 | 1.0必须做轻量版，避免大而全。 |
| 数据接入层 | API接口、文件导入、数据库同步、人工录入、第三方平台对接。 | 兼容当前多源异构数据。 |
| 数据源与感知层 | 无人机、车辆定位、抬斗识别、摄像头AI、边缘计算、称重设备、传感器、设计资料。 | 明确数据来源、频率、格式和可信度。 |
| 基础设施层 | 服务器、数据库、对象存储、内网部署、安全防护、备份恢复、国产化适配。 | 保证系统可部署、可运维、可安全运行。 |

### 4.1 架构设计原则

1. 前端不直接绑定硬件厂商接口，统一通过数据服务层调用。
2. 数据接入和业务功能解耦，便于后续更换设备、增加数据源或复制到其他项目。
3. 1.0阶段保留人工导入与人工确认机制，避免因单一硬件接口未打通导致系统停摆。
4. 架构层预留国产化、私有化部署、多项目管理和局级平台集成接口。
5. 系统架构围绕时间进度、空间流向、现场修正、调配闭环组织服务。

### 4.2 数据底座与接入原则

1. 1.0阶段以项目级轻量底座为主，支撑接入、清洗、编码、坐标、追溯，不做重型大数据平台。
2. 数据接入按P0/P1/P2分层推进，P0优先接设计量、实际量、车辆定位、称重/运输记录、无人机测量结果；P1接抬斗识别、摄像头AI、设备在线状态、道路矩阵；P2预留复杂传感器、全量视频、全量点云。
3. 主数据重点围绕项目、标段、料场、渣场、中转场、填筑区、道路、车辆、设备和调配方案。
4. 指标口径聚焦设计开挖量、实际开挖量、实际填筑量、直接上坝率、中转量、弃渣量和进度偏差。
5. 坐标与空间标准统一处理WGS84、CGCS2000、施工坐标、BIM坐标和三维引擎坐标。
6. 数据原则仍然是样例先行、导入兜底、接口预留、逐步适配。

| 优先级 | 数据类型 | 说明 |
|---|---|---|
| P0 | 基础资料、设计量、实际量、车辆定位、称重/运输记录、无人机测量结果。 | 支撑系统基本可用和核心展示。 |
| P1 | 抬斗识别、摄像头AI事件、设备在线状态、道路矩阵。 | 支撑运输监管、异常识别和调配辅助。 |
| P2 | 复杂传感器、全量视频、全量点云在线处理、多厂家自动适配。 | 后续迭代，避免1.0范围失控。 |

## 五、核心应用闭环

前文已明确1.0范围、时空关系和总体架构，本章不再重复罗列功能模块，重点说明系统在演示和试运行阶段如何串成可理解、可验证、可复核的业务闭环。

### 5.1 1.0演示闭环路径

1. 进入项目驾驶舱，展示项目整体土石方平衡状态。
2. 进入三维一张图，展示开挖区、填筑区、中转场、弃渣场、道路和车辆。
3. 点击料场或填筑区，查看设计量、实际量、剩余量、进度和关联预警。
4. 查看车辆运输轨迹，关联称重记录和抬斗识别事件。
5. 进入动态调配，基于当前数据输出基础调配建议。
6. 进入预警中心，查看并处置料源不足、容量超限、车辆异常或进度滞后问题。
7. 生成日报、周报或阶段性汇报报表，形成可追溯成果。

### 5.2 设计参考与施工实测修正

系统应将原设计资料、地质资料、钻孔资料、土石方平衡规划等作为“设计参考模型”的数据来源，将现场揭露、无人机测量、称重运输、人工确认等实际发生数据作为“施工实测模型”的数据来源。两套模型应并行管理、可对比、可修正，设计数据不直接替代现场实际数据。

| 模型类型 | 数据来源 | 主要作用 | 1.0边界 |
|---|---|---|---|
| 设计参考模型 | 设计资料、钻孔资料、地质资料、原始地形、土石方平衡规划。 | 作为初始基准，展示预测料源、可用料、弃料及调配关系。 | 支持数据录入、导入、空间绑定和初始可视化。 |
| 施工实测模型 | 现场揭露、无人机测量、称重数据、车辆运输、人工确认。 | 反映实际料源、工程量、进度和调配变化。 | 支持阶段性更新、人工/半自动修正、设计与实际对比。 |
| 偏差分析模型 | 设计参考模型与施工实测模型对比结果。 | 识别可用料不足、超量、地质偏差、进度偏差和运输异常。 | 形成基础偏差统计和预警，后续增强智能分析。 |

## 六、技术路线与选型建议

技术选型遵循成熟稳定、易交付、可私有化、可国产化适配、可扩展、易维护的原则。最终技术栈结合甲方环境、既有系统、部署要求和团队技术能力确认。

| 技术方向 | 建议路线 | 说明 |
|---|---|---|
| 前端管理端 | Vue / React + 组件库 + 图表库。 | 用于后台管理、数据报表、业务流程操作。 |
| 可视化体系 | Web端ECharts + Cesium/Mapbox/OpenLayers + UE5高品质三维时空可视化引擎。 | Web端负责日常业务、GIS底图、点位、道路、报表和轻量展示；UE5负责重点三维场景、施工时序、车辆轨迹、地质模型变化和汇报展示，二者共享统一数据服务。 |
| 后端服务 | Java/Spring Boot 或 .NET/Node 等成熟服务框架。 | 根据团队和项目要求确认，重点是稳定与可维护。 |
| 数据库 | PostgreSQL/PostGIS 或国产数据库适配路线。 | 同时考虑空间数据、业务数据和国产化适配。 |
| 对象存储 | 本地文件服务 / MinIO / 私有化对象存储。 | 管理无人机成果、点云、模型、附件、截图。 |
| 接口服务 | REST API / WebSocket / 定时任务 / 消息队列按需引入。 | 车辆定位可用WebSocket或定时轮询，1.0不必过度复杂。 |
| 数据处理 | ETL脚本、导入校验、坐标转换、数据清洗服务。 | 形成轻量治理能力。 |
| 部署方式 | 私有化部署、内网访问、备份恢复、日志审计。 | 满足工程项目现场环境和数据安全要求。 |

### 6.1 不建议的技术路线

1. 不建议1.0阶段建设复杂大数据平台、数据湖和全量流式处理平台。
2. 不建议1.0阶段自研完整视频平台，优先接入AI识别事件和第三方视频平台链接。
3. 不建议1.0阶段做全自动点云在线算量和三维模型实时重建，可先接入测量成果和工程量结果。
4. 不建议把动态调配做成不可解释黑盒算法，先做规则型、可人工确认的辅助调配。
5. 不建议为了追求前端炫酷效果牺牲功能落地。Web端和UE5端均应服务于业务主线、时空关系、调配过程和风险识别。

### 6.2 国产化、私有化与安全

| 适配方向 | 1.0建议 | 后续增强 |
|---|---|---|
| 国产服务器 | 支持普通x86环境，预留国产CPU适配验证计划。 | 鲲鹏、飞腾等环境适配测试。 |
| 国产操作系统 | 优先采用跨平台部署架构。 | 统信、麒麟等系统部署验证。 |
| 国产数据库 | 数据库访问层避免强绑定单一数据库能力。 | 适配达梦、人大金仓等国产数据库。 |
| 国产浏览器 | 前端遵循标准Web能力。 | 适配常见国产浏览器和内网浏览器。 |
| 国产中间件 | 减少对不可控商业中间件依赖。 | 根据项目要求替换。 |

安全与权限设计包括用户身份认证、角色权限控制、数据权限控制、操作审计、数据备份和内网部署。涉及司机定位、人员影像、车辆轨迹等数据时，应按最小必要原则进行采集、展示和留存。

## 七、实施计划与组织协同

建议按照100天左右规划1.0版本交付周期，并设置阶段性成果节点。该周期包含前端、后端、数据库、GIS/三维展示、数据导入、基础调配、预警报表、权限管理、联调测试和部署验收准备等工作。每个阶段都要有可展示成果，避免所有功能集中到最后联调。

### 7.1 实施阶段

| 阶段 | 周期 | 主要工作 | 阶段成果 |
|---|---|---|---|
| 阶段一：需求与架构确认 | 第1-10天 | 需求访谈、数据样例收集、硬件接口摸底、1.0边界确认、总体架构确认。 | 需求确认记录、功能清单、架构图、数据清单、接口清单、风险清单、1.0功能边界确认表。 |

| 阶段七：测试与部署准备 | 第86-95天 | 系统测试、性能优化、Bug修复、部署环境准备、演示汇报。 | 测试记录、问题修复清单、部署包、演示数据、上线前预验收版本。 |

### 7.3 现场调研与业务确认机制

现场调研成果应形成《现场调研记录》《业务流程确认表》《数据接口清单》《1.0功能边界确认表》《风险与待验证问题清单》，并作为后续开发、验收和范围控制的重要依据。

## 九、风险控制与配合条件

### 9.1 主要风险与控制措施

| 风险 | 影响 | 控制措施 |
|---|---|---|
| 硬件接口不确定 | 影响数据接入进度。 | 前期收集接口文档和样例；接口接入、导入、人工录入并行。 |
| 数据质量不稳定 | 影响统计、预警和调配可信度。 | 建立数据质量校验、可信度标记和人工确认机制。 |
| 需求范围扩大 | 影响周期和成本。 | 书面固化1.0功能边界，新增需求进入迭代清单。 |
| 坐标模型错位 | 影响三维展示和空间分析。 | 前期完成坐标基准确认和模型对齐测试。 |
| 数据不完整或关键约束缺失 | 缺少道路、料质、时间窗口、堆场容量等关键数据，影响算法和报表可信度。 | 先完成数据摸底和样例数据验证，明确缺失项、替代策略和后续补齐责任。 |
| BIM/GIS资料不足 | 模型无法直接接入、坐标或工程部位不匹配，影响数字孪生驾驶舱和空间调配展示。 | 启动阶段完成资料摸底，明确轻量化建模、格式转换、二维GIS过渡展示或分阶段补齐方案。 |
| 首期周期偏紧 | 需求确认、数据摸底、接口联调时间过长，压缩实际开发窗口。 | 聚焦1.0核心闭环，先形成可演示、可复核成果；复杂能力进入后续版本。 |
| 算法预期过高 | 希望系统直接替代人工调度或输出完全自动决策。 | 明确算法定位为辅助决策和方案复核，优先可解释、可修正、可人工确认。 |
| 知识产权边界不清 | 影响双方后续合作。 | 合同中明确项目定制成果、既有组件、第三方库和开源组件边界。 |
| 现场业务配合不足 | 影响需求准确性和数据获取。 | 明确甲方业务负责人、项目部、硬件厂商和测量单位配合机制，固定开展数据核对和需求确认工作。 |
| 复合能力协同风险 | 项目同时涉及工程业务、三维展示、数据治理、调配算法、现场硬件和系统开发，单一能力强不代表整体能落地。 | 建立甲方业务专家、技术负责人、开发团队、测量/硬件单位、算法支持人员协同机制，按里程碑验证。 |

### 9.2 甲方需提供资料

项目启动阶段需要提供的资料，控制在基础资料、设计/测量成果、设备与接口样例、现场工况说明四类即可，其他内容按推进节奏补齐。

4. 基于料源、道路、容量和进度约束的土石方调配辅助方法。

6. 基于时空关系的土石方平衡过程管理与调配辅助方法。
